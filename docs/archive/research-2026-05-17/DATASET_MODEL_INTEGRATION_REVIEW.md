# Dataset, Model, and Integration Review

Date: 2026-05-17

## Applicability

This project is not an ML product. It is a static public portfolio. There is no current need for hosted model inference, AI runtime features, user tracking, or sensitive data processing.

The project does have a meaningful data and integration layer:

- Public GitHub repository metadata.
- Star counts.
- Latest release metadata.
- README excerpts.
- Project catalog records.
- Screenshots for live apps.
- Changelog and release history.
- RSS and generated pages.

## Existing Data Sources

| Source | Current storage | Notes |
|---|---|---|
| Project catalog | `src/data/projects.ts` | Manual, typed, not schema-validated beyond TypeScript shape. |
| Curated highlights | `src/data/curated.ts` | Manual narratives and groupings. |
| GitHub stars/stats | ignored `src/data/_stars.json`, `_stats.json` | Generated by `scripts/fetch-stars.mjs`; can become stale. |
| GitHub repo metadata | ignored `src/data/_meta.json` | Generated cache. |
| GitHub releases | ignored `src/data/_releases.json` | Generated cache. |
| README excerpts | ignored `src/data/_readmes.json` | Generated cache, parsed and sanitized at build time. |
| Screenshots | `public/screenshots/*.jpg` | Tracked assets, manually refreshed. |

## External Integrations

- GitHub REST/API through `scripts/fetch-stars.mjs`.
- GitHub Pages deployment through Actions.
- Browser-side GitHub API enhancement in `public/scripts/main.js`.
- RSS consumers for generated feeds.

## Opportunities

### 1. Treat catalog data as a checked dataset

Add schema validation and audit reports:

- Required fields.
- Unique slugs and names.
- Valid categories/languages.
- Valid public GitHub URLs.
- Screenshot presence for live apps.
- Explicit skip exceptions for public repos not listed.
- Explicit privacy review flags.

### 2. Add machine-readable public feeds

Generate static JSON:

- `projects.json`
- `releases.json`
- `catalog-stats.json`

These should include schema version and freshness metadata.

### 3. Improve static search

Use a static search index from:

- Project names/descriptions.
- Tags/categories/languages.
- README excerpts.
- Project detail page text.

Candidate tools:

- Pagefind.
- MiniSearch.
- Fuse.js.
- Lunr.

### 4. Add lightweight offline classification

Potential non-runtime uses:

- Suggest categories for newly public repos.
- Detect near-duplicate descriptions.
- Suggest related projects.
- Evaluate search relevance against a small checked query set.

Any model-assisted processing should be offline, reproducible enough for review, and not require visitor data or hosted inference.

## Evaluation Ideas

Create a small search relevance fixture:

| Query | Expected hits |
|---|---|
| `dicom` | DICOM and imaging-safe public entries only. |
| `android backup` | Android backup/restore projects. |
| `chrome extension` | Extension projects including `AI-Usage_Tracker` after review. |
| `screenshot crop` | Screenshot/image utilities. |
| `terminal` | CLI and terminal tools. |

Create a catalog audit fixture:

- Live GitHub repos minus catalog entries.
- Catalog entries minus live public repos.
- Public forks included in catalog.
- Public repos intentionally skipped.
- Privacy-sensitive repos requiring explicit review.

## Non-Goals

- No hosted vector database.
- No visitor behavior tracking.
- No dynamic recommendation service.
- No model-generated public claims without source evidence.
- No automated publication of private or sensitive repos.

## Why This File Is Thin

The project has data and integration concerns, but not model-serving concerns. The highest-value work is deterministic catalog validation, static search, and generated public metadata. Model usage is optional and should remain offline unless a future product direction changes.
