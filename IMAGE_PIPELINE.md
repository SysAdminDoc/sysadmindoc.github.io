# Image Pipeline

Status date: 2026-05-17
Site version: v0.16.13

This file is the source of truth for portfolio visual assets: generated social cards, live-app screenshot masters, smaller card thumbnails, and remote README images rendered on project pages.

## Social Cards

Project social images are generated by [src/pages/og/[slug].png.ts](src/pages/og/%5Bslug%5D.png.ts). The route:

- Builds one static route per unique featured, live-app, and catalog slug.
- Uses Satori plus Resvg to render a 1200x630 card.
- Returns `Content-Type: image/png`.
- Keeps immutable cache headers because the URL is slug-stable and regenerated at build time.

[src/layouts/Base.astro](src/layouts/Base.astro) publishes `og:image`, `og:image:type`, `og:image:width`, `og:image:height`, `og:image:alt`, and matching Twitter card metadata. Keep these cards PNG-compatible for social platforms.

## Screenshots And Thumbnails

Live app visual assets are deliberately split:

- Full screenshot masters: `public/screenshots/<slug>.jpg`
- Card thumbnails: `public/screenshots/thumbs/<slug>.jpg`

[scripts/capture-screenshots.mjs](scripts/capture-screenshots.mjs) captures deployed live apps with Playwright and now writes both artifacts from the same captured page surface. Full masters are retained for project detail pages. Thumbnails are 640x400 JPEG files generated separately with Sharp at lower quality for homepage/live-grid cards.

[src/components/LiveCard.astro](src/components/LiveCard.astro) uses the thumbnail path. [src/pages/projects/[slug].astro](src/pages/projects/%5Bslug%5D.astro) uses the full master on project detail pages.

Current audited inventory:

- Live apps checked: 22
- Full screenshot total: 1595.2 KB
- Thumbnail total: 230.9 KB
- Thumbnail ratio: 14.5% of full screenshot bytes
- Largest full screenshot: `StormviewRadar` at 273.9 KB
- Largest thumbnail: `StormviewRadar` at 37.3 KB

## README Images

Project pages render sanitized README excerpts from generated GitHub cache data. Relative Markdown and raw HTML image references are resolved to raw GitHub asset URLs, and relative links are resolved to GitHub blob URLs. Rendered README images are lazy-loaded and decoded asynchronously.

Do not proxy or commit arbitrary README images into this repo. They belong to the source project and can change independently of this portfolio.

## Astro Image Tooling Decision

Astro's image guide recommends keeping local images in `src/` when Astro should transform, optimize, and bundle them, while `public/` files are copied and served as-is. The same guide notes that remote image processing requires authorized sources, and native `<img>` output is not processed by Astro.

Decision for v0.16.13:

- Keep OG cards on the Satori/Resvg endpoint. It already emits static PNG social-card routes and does not benefit from an Astro image migration.
- Keep live screenshot masters in `public/screenshots/` because they are intentionally direct public assets used by static pages and manual review.
- Use Sharp in repo scripts for derived thumbnails because that keeps card compression explicit, repeatable, and independent of GitHub Pages runtime behavior.
- Do not migrate remote README images to `astro:assets`; they come from many GitHub repositories and would require remote-source authorization plus caching decisions that add build fragility.

Revisit Astro image components later only for stable local assets that can move under `src/` without changing public URLs or the GitHub Pages deployment model.

## Commands

```bash
npm run capture-screenshots # recapture live-app masters and thumbnails
npm run screenshots:thumbs  # regenerate thumbnails from tracked masters
npm run images:audit        # validate screenshot, thumbnail, and OG invariants
npm run check               # includes data, asset, image, and Astro checks
npm run build               # includes data, asset, image, Astro, and Pagefind steps
```
