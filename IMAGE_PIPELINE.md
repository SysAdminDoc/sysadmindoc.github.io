# Image Pipeline

Status date: 2026-06-04
Site version: v0.18.3

This file is the source of truth for portfolio visual assets: generated social cards, live-app screenshot masters, smaller card thumbnails, and remote README images rendered on project pages.

## Social Cards

Project and key interior-page social images are generated by [src/pages/og/[slug].png.ts](src/pages/og/%5Bslug%5D.png.ts). The route:

- Builds one static route per unique featured, live-app, catalog, and registered interior-page slug.
- Uses [src/data/interior-og-pages.ts](src/data/interior-og-pages.ts) as the shared metadata source for `/uses/`, `/resume/`, `/search/`, `/timeline/`, `/archive/`, `/now/`, `/healthcare-it/`, and `/releases/`.
- Uses Satori plus Resvg to render a 1200x630 card.
- Returns `Content-Type: image/png`.
- Keeps bounded generated-image cache headers because `/og/*.png` routes are slug-stable but not hashed.

[src/layouts/Base.astro](src/layouts/Base.astro) publishes `og:image`, `og:image:type`, `og:image:width`, `og:image:height`, `og:image:alt`, and matching Twitter card metadata. Keep these cards PNG-compatible for social platforms.

## Screenshots And Thumbnails

Live app visual assets are deliberately split:

- Full screenshot masters: `public/screenshots/<slug>.jpg`
- Stable public card thumbnails: `public/screenshots/thumbs/<slug>.jpg`
- Astro-managed card thumbnail inputs: `src/assets/screenshots/thumbs/<slug>.jpg`

[scripts/capture-screenshots.mjs](scripts/capture-screenshots.mjs) captures deployed live apps with Playwright and now writes the public full master, the stable public thumbnail, and the matching Astro asset thumbnail from the same captured page surface. Full masters are retained for project detail pages and direct public URLs. Thumbnails are 640x400 JPEG files generated with Sharp at lower quality for homepage/live-grid cards.

[src/components/LiveCard.astro](src/components/LiveCard.astro) imports the `src/assets/screenshots/thumbs/` copy and renders it through Astro's `<Picture>` component with AVIF/WebP sources, width-based srcsets, and a JPEG fallback. [src/pages/projects/[slug].astro](src/pages/projects/%5Bslug%5D.astro) continues to use the full public master on project detail pages so stable screenshot URLs and `/projects.json` do not break.

Current audited inventory:

- Live apps checked: 22
- Full screenshot total: 1595.2 KB
- Thumbnail total: 230.9 KB
- Astro asset thumbnail inputs: 22
- Thumbnail ratio: 14.5% of full screenshot bytes
- Largest full screenshot: `StormviewRadar` at 273.9 KB
- Largest thumbnail: `StormviewRadar` at 37.3 KB

## README Images

Project pages render sanitized README excerpts from generated GitHub cache data. Relative Markdown and raw HTML image references are resolved to raw GitHub asset URLs, and relative links are resolved to GitHub blob URLs. Rendered README images are lazy-loaded and decoded asynchronously.

Do not proxy or commit arbitrary README images into this repo. They belong to the source project and can change independently of this portfolio.

## Astro Image Tooling Decision

Astro's image guide recommends keeping local images in `src/` when Astro should transform, optimize, and bundle them, while `public/` files are copied and served as-is. The same guide notes that remote image processing requires authorized sources, and native `<img>` output is not processed by Astro.

Decision as of v0.18.3:

- Keep OG cards on the Satori/Resvg endpoint. It emits static PNG social-card routes for both project and registered interior-page slugs and does not benefit from an Astro image migration.
- Keep live screenshot masters in `public/screenshots/` because they are intentionally direct public assets used by static pages and manual review.
- Keep stable thumbnail copies in `public/screenshots/thumbs/` for JSON/public URL compatibility, but render homepage/live-grid card thumbnails from `src/assets/screenshots/thumbs/` through Astro `<Picture>` so the built site emits AVIF/WebP srcsets and hashed image URLs.
- Use Sharp in repo scripts for derived thumbnail inputs because that keeps card compression explicit, repeatable, and independent of GitHub Pages runtime behavior.
- Do not migrate remote README images to `astro:assets`; they come from many GitHub repositories and would require remote-source authorization plus caching decisions that add build fragility.

Revisit Astro image components later only when another stable local asset can move under `src/` without changing public URLs or the GitHub Pages deployment model.

## Commands

```bash
npm run capture-screenshots # recapture live-app masters, public thumbs, and Astro thumb inputs
npm run screenshots:thumbs  # regenerate public thumbs and Astro thumb inputs from tracked masters
npm run images:audit        # validate screenshot, thumbnail, and OG invariants
npm run check               # includes data, asset, image, and Astro checks
npm run build               # includes data, asset, image, Astro, and Pagefind steps
```
